**三语 EPUB 小说翻译阅读器**

需求分析与技术选型文档（MVP 导向）

---

文档目的：在“不重复造轮子”的前提下，明确该产品的业务目标、功能边界、关键技术方案、现成框架/库选型，以及可落地的
MVP 实施路径。

目标产品：用户上传中文 / 日文 / 英文小说（优先
EPUB），系统按章节懒翻译为三语版本，并在 Web
阅读器中进行对照阅读、同步学习与后续导出。

*建议发布日期：2026-04-05*

# 1. 项目背景与产品定位

该项目的核心不是“做一个通用翻译器”，而是做一个面向小说阅读学习场景的三语
AI
阅读产品：上传一本电子书后，系统保留原书章节结构，按阅读进度逐章生成中文、日文、英文三个版本，并在阅读器中同步展示。

与一次性整本翻译相比，本产品更适合采用“首章预翻译 + 当前章即时翻译 +
下一章预取”的懒处理模式。这样可以显著降低 token
成本、缩短首屏等待时间，并把失败重试与断点恢复控制在章节粒度。

# 2. 建设目标

**•** 上传 EPUB/TXT 后，自动识别源语言（中文、日文、英文其一）。

**•** 解析目录、章节、段落结构，保留尽可能多的基础排版信息。

**•** 按章节触发三语翻译任务，并将结果以段落级数据对齐保存。

**•** 在 Web 端提供类似 EPUB 的阅读体验，支持单语 / 双语 / 三语切换。

**•** 支持后续导出：至少 HTML ZIP / JSON，后续再支持三语 EPUB。

**•**
必须支持断点恢复、失败重试、章节状态可见，不因单次任务失败而整本重来。

# 3. 非目标（MVP 暂不纳入）

**•** 不做出版级 100% 排版还原；复杂 CSS、竖排、ruby 注音等先不承诺。

**•** 不做整本一次性翻译完成后再进入阅读。

**•** 不做复杂句级对齐、全文术语库协同编辑、多人协作批注。

**•** 不做本地离线大模型推理；模型侧优先采用云端 API。

**•** 不做开放式 UGC 社区分发，避免版权与审核复杂度过早上升。

# 4. 目标用户与典型场景

| **角色**        | **核心诉求**             | **典型行为**                          | **产品价值**                   |
|-----------------|--------------------------|---------------------------------------|--------------------------------|
| 语言学习者      | 想边读小说边对照理解     | 上传原文、切换双语/三语、点击段落对比 | 减少查词与来回切换成本         |
| 轻小说/网文读者 | 想快速读懂日文或英文原作 | 优先看原文+中文辅助，逐章阅读         | 更快进入内容，不必等待整本完成 |
| 高级学习者      | 关心表达差异与语感       | 对照原文、英文、中译，观察句法和语气  | 更适合精读与模仿写作           |

结论：该产品首先是“阅读器”，其次才是“翻译器”。因此阅读体验、状态感知和渐进式生成，比一次性翻译完更重要。

# 5. 核心需求分析

## 5.1 文件导入与解析

**•** 支持 EPUB（优先）与 TXT（次优）上传。EPUB 为主，因为其天然具备目录、章节和 XHTML 结构。

**•**
上传后先完成元数据读取：书名、作者、目录、章节顺序、封面（可选）、章节资源路径。

**•** 解析阶段需要把书内 XHTML/HTML 正文拆成“章节 -\>
段落/块”的内部统一结构，供后续翻译和阅读器渲染使用。

**•**
解析程序应保留基础标签语义：标题、段落、粗体、斜体、分隔符、图片占位等；复杂排版后续降级处理。

## 5.2 翻译与任务编排

**•** 上传完成后默认预翻译第 1 章（可选前 2 章），让用户尽快开始阅读。

**•** 用户打开第 N 章时，若尚未生成三语内容，则触发该章翻译任务。

**•** 用户阅读第 N 章时，后台预取第 N+1 章，降低翻页等待。

**•**
章节内部不能一次把整章文本直接丢给模型；应按段落或小批次执行，以规避上下文超限并提升恢复能力。

**•** 每个段落/批次必须有状态：pending / processing / done /
failed。章节状态由子任务聚合得出。

## 5.3 阅读器

**•** 支持原文单语、原文+译文双语、三语对照三种模式。

**•** 支持桌面端三栏布局，移动端折叠/切换布局。

**•** 支持章节目录跳转、阅读进度保存、字号与主题切换。

**•** 支持段落级联动：点击原文段落，高亮对应中文与英文译文。

**•** 优先实现“结构化阅读器”，不要求 100% 复刻所有 EPUB 阅读器行为。

## 5.4 导出与资产归属

**•** MVP 先支持导出 JSON 与 HTML ZIP；导出三语 EPUB 放到第二阶段。

**•**
用户已生成的翻译内容应归属用户本人，可下载，不应被限制只能在线阅读。

**•** 导出需要保留章节结构、语言标识和基础目录信息；复杂样式允许降级。

# 6. 为什么不建议“纯前端”

阅读器本身可以高度前端化，但完整产品不建议做成纯前端。原因在于：模型密钥暴露风险、长任务编排、失败重试、进度持久化、导出生成、版权与存储策略都更适合放在轻后端或
serverless 层。

**•** 前端负责：上传交互、阅读体验、状态展示、用户偏好。

**•**
后端负责：鉴权、文件存储、模型调用、章节任务队列、断点恢复、导出生成。

# 7. 技术选型：框架与库（优先选成熟现成方案）

## 7.1 前端框架对比

| **方案**              | **技术基础** | **优势**                                                                                                                                                       | **限制**                                                                      | **结论**    |
|-----------------------|--------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------|-------------|
| Next.js               | React        | 生态成熟，适合全栈 Web；App Router、Server Functions/Route Handlers 适合把轻后端与前端放在同一仓库；和 Clerk、Supabase、Vercel AI SDK、react-reader 组合顺手。 | React 心智负担略高；服务端/客户端边界需要规范。                               | 推荐首选    |
| Nuxt                  | Vue          | Vue 语法亲和；Nuxt 自带全栈能力，SSR/SSG 体验好。                                                                                                              | EPUB 阅读器与三语对照的现成生态略弱于 React 组合；若团队已有 Vue 经验则可选。 | 可选替代    |
| 纯 React / 纯 Vue SPA | 前端单页     | 上手快，演示简单。                                                                                                                                             | 不利于鉴权、任务编排和模型安全接入；正式产品不推荐。                          | 仅适合 Demo |

## 7.2 阅读器与 EPUB 相关库推荐

**•** epub.js：浏览器内渲染 EPUB 的成熟基础库，适合阅读、分页、导航、定位和 CFI 能力。

**•** react-reader：基于 epub.js 的 React 封装，适合快速接入现成阅读器壳子，减少从零搭建成本。

**•** JSZip：可用于在浏览器或服务端读取 ZIP 结构，适合处理 EPUB 容器与 HTML ZIP 导出。

**•** fast-xml-parser / @xmldom/xmldom / Cheerio：适合服务端解析 OPF、NCX、XHTML 与正文节点。

建议做法：阅读层使用现成阅读器（epub.js /
react-reader），而“翻译与对齐”使用独立的结构化内容管线；不要强依赖阅读器内部
DOM 作为翻译数据源。

## 7.3 鉴权、数据库、存储、任务系统推荐

**•** 鉴权：Clerk（接入快、预制 UI 强）或 Supabase
Auth（数据库一体化更强）。

**•** 数据库：Supabase
Postgres，适合保存图书、章节、翻译任务、阅读进度、导出记录。

**•** 对象存储：Supabase Storage / S3 / R2，用于保存原始
EPUB、章节中间产物与导出文件。

**•** 任务编排：Inngest 或 Trigger.dev。该项目需要长任务、自动重试、并发控制和可观测性，采用现成
durable workflow 明显优于手写队列。

## 7.4 模型接入层推荐

**•** 做 Provider 抽象，不把模型写死；至少预留
OpenAI、Claude、Gemini、千帆/文心等适配层。

**•** 如果主技术栈选 Next.js/TypeScript，可考虑 Vercel AI SDK 作为统一模型调用封装，再补自定义 provider。

**•**
翻译接口要支持：源语言、目标语言、上下文片段、术语表、风格提示、重试和幂等键。

# 8. 推荐技术栈（建议版本路线）

**推荐主栈：**Next.js + React + TypeScript + Tailwind CSS +
epub.js/react-reader + Clerk 或 Supabase Auth + Supabase
Postgres/Storage + Inngest/Trigger.dev + Provider 抽象的 LLM 接入层。

推荐原因：同一仓库即可覆盖前端、API、轻后端与后台任务；React 侧的 EPUB 集成选择更顺手；后续也更容易接入导出、支付、限额与运营功能。

# 9. 系统架构建议

**•** Web App：负责上传、书库、阅读器、任务状态展示与导出下载。

**•** API/Server
Functions：负责上传签名、书籍创建、章节请求、翻译任务触发、导出触发。

**•** Background Worker：负责 EPUB 解析、章节拆分、翻译小批次执行、失败重试、结果合并。

**•** Database：保存 Book / Chapter / Paragraph / Translation / Task /
ReadingProgress 等表。

**•** Storage：保存原始文件、提取后章节、导出产物。

# 10. 数据模型（MVP）

| **实体**        | **关键字段**                                    | **说明**                   |
|-----------------|-------------------------------------------------|----------------------------|
| Book            | id, title, source_lang, file_path, status       | 一本书的元数据与总状态     |
| Chapter         | id, book_id, index, title, source_html, status  | 一章原始内容及处理状态     |
| Paragraph       | id, chapter_id, seq, source_text, source_markup | 章节内统一结构化片段       |
| Translation     | paragraph_id, lang, text, status, model, tokens | 段落级翻译结果，可独立重试 |
| ReadingProgress | user_id, book_id, chapter_index, position       | 用户阅读位置与偏好         |

# 11. 关键流程设计

## 11.1 上传到首章可读

**•** 用户上传 EPUB -\> 创建 Book 记录。

**•** 后台解析 container.xml / OPF / TOC / spine -\> 建立章节索引。

**•** 抽取第 1 章结构化内容 -\> 按段落写入 Paragraph。

**•** 触发第 1 章 zh/en（或缺失两语）翻译任务。

**•** 阅读器先展示目录和原文；译文完成后无刷新或轻刷新补齐。

## 11.2 按阅读进度懒翻译

**•** 打开第 N 章时，若翻译不存在则创建章节任务。

**•** 当前章翻译优先级最高；下一章预取优先级次之。

**•** 失败段落仅重试失败项，不重跑整个章节。

**•** 每章允许显示“原文已就绪 / 中文生成中 / 英文生成中”等细粒度状态。

# 12. MVP 范围建议

**•** 文件格式：EPUB、TXT。

**•** 语言：中 / 日 / 英三语。

**•** 阅读模式：单语、双语、三语。

**•** 任务模式：首章预翻译 + 当前章即时翻译 + 下一章预取。

**•** 导出：JSON、HTML ZIP。

**•** 账号：邮箱验证码 + 第三方登录（二选一即可）。

不建议在 MVP 第一阶段支持整本导出三语
EPUB、复杂全文术语库、精细句对齐、朗读、生词本等高级能力。

# 13. 风险与应对

**•** 风险 1：EPUB 原始排版过于复杂。应对：MVP
只承诺保留章节结构和基础样式，不承诺 100% 出版级还原。

**•** 风险 2：长篇小说 token
成本高。应对：按章节懒翻译、限制免费额度、缓存已完成章节。

**•** 风险 3：多章分批翻译的人名术语不一致。应对：后续增加书级 glossary
和章节回写校正。

**•** 风险
4：模型调用失败或额度不足。应对：任务持久化、分段状态、幂等重试、断点续跑。

**•** 风险
5：版权问题。应对：默认私有、限制公开分享、支持删除源文件与导出资产。

# 14. 最终建议

**建议结论：**采用 Next.js 为主的“前端 + 轻后端 + durable
任务系统”方案，阅读器直接复用 epub.js / react-reader，不自己从零实现
EPUB
渲染；翻译管线则独立成结构化章节/段落处理系统。这样既能保证开发速度，也能兼顾后续扩展、成本控制和可恢复性。

如果团队本身更熟 Vue/Nuxt，也可以平移方案；但在“现成阅读器生态 + 统一
TypeScript 工程 + 模型 SDK 适配 + 快速出 MVP”这一组合上，Next.js
路线更顺手。
